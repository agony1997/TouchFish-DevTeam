# dev-team v1.1.0 設計決策紀錄

> 來源：2026-03-03 挑戰者審查對話（Challenger Review Session）
> 角色：Claude 擔任挑戰/質疑者，針對 dev-team v1.0.0 進行系統性批判
> 結論：產出 v1.1.0 修改規格，涵蓋四項主要變更 + 兩項附帶改進

## 一、審查背景與目標

使用者要求 Claude 以「挑戰者」身份，從四個面向質疑 dev-team v1.0.0：

1. 該專案技能的存在用途
2. 各階段（P0-P5）的必要性
3. 是否該跟 TouchFish-Skills 的技能綁定
4. 是否該耦合 OpenSpec & superpowers

Claude 先透過 sub-agent 完整探索了 TouchFish-DevTeam 和 TouchFish-Skills 兩個專案的結構，
再基於探索結果提出系統性質疑。

## 二、v1.0.0 識別的問題

### 2.1 技能存在價值的質疑

| # | 質疑 | 嚴重程度 |
|---|------|---------|
| 1 | Claude Code 本身就有 Agent tool + TeamCreate，dev-team 的 orchestration 開銷可能超過品質收益 | 高 |
| 2 | 一次完整流程可能 spawn 18 個 agent 實例（TL + 5 test-agents + 5 workers + 5 qa-tasks + 1 qa-global + 1 delivery-sub），Token 成本巨大 | 高 |
| 3 | 「分離測試」（Opus 寫測試、Sonnet 寫實作）的優勢在嚴格 TDD 下被削弱 — 因為測試先寫時，自我驗證偏差本就不會發生 | 中 |

### 2.2 各階段必要性的質疑

| Phase | 質疑 | 結論 |
|-------|------|------|
| P0 Reconnaissance | 80% 內容是偵測外部 plugin，一個 Glob 就能做完，是否值得獨立為一個 Phase？ | 可與 P1 合併 |
| P1 Requirements | 過度儀式化 — 同時產出 PLAN.md（LLM 格式）和 plan-summary.md（人類格式），且 TaskCreate description 已包含大部分資訊 | PLAN.md 有其文件至上的價值（見下方化解） |
| P2 API Contract | 只有需要 API endpoint 的任務才觸發，大部分時候 skip；且 Contract 修訂流程（kill Workers → respawn）成本極高 | 保留但適用場景窄 |
| P3 Development | 核心階段。但 QA FAIL → 新 Worker（新 context）的 fix loop 設計脆弱 — 新 Worker 不知道前一個為什麼 fail | 保留，但需要改進 |
| P4 Global Review | 如果 P1 任務拆解做得好（file scope 不重疊），跨任務一致性問題根本不該出現。P4 是在為 P1 的失誤買保險 | 保留（防禦性設計） |
| P5 Delivery | DELIVERY.md 有 8 個 section，其中 Agent Metrics 與 code review audit trail 無關 | 移除 Agent Metrics |

### 2.3 外部 plugin 耦合的質疑

dev-team 的設計原則 P2 宣稱「Plugin-independent + detection-enhanced」—— 用 Glob 偵測
superpowers-tdd、reviewer、explorer、openspec，找到就增強，找不到就獨立運行。

**三個核心問題：**

1. **測試矩陣爆炸** — 4 個可選 plugin 產生 2^4 = 16 種組合，無法保證所有組合正常運作
2. **無版本控制** — superpowers 改了 TDD skill 介面，dev-team 不會知道
3. **職責邊界模糊** — 有 superpowers 是「嚴格 TDD」，沒有就是「非嚴格 TDD」，品質差異無法量化

**結論：「可選偵測」看似最靈活，實際上是在迴避決策。**

### 2.4 與 TF-Skills 拆分的質疑

dev-team 從 TF-Skills 拆為獨立 repo，理由是「完整開發框架 vs pipeline 單元」。但：

- md-to-code 也有 Agent Teams（backend-dev + frontend-dev），也 spawn 多個 teammate 並行開發
- 拆分後使用者需同時安裝兩個 repo，且版本不匹配時無保障

## 三、使用者提供的期望與質疑化解

使用者在耦合議題上**選擇方案 A（完全獨立）**，並提供六項期望：

| # | 期望 | 化解了哪些質疑 |
|---|------|--------------|
| 1 | 增加並行度 | — |
| 2 | 文件至上，不信任 LLM 的 context/memory | 化解「PLAN.md 存在價值」質疑 — PLAN.md 是讓 agent 從檔案重建 context |
| 3 | 以文件規範 AI 產出，所有 AI coding 產物必須保持可控制 | — |

| 5 | LLM-native 格式是期望增加 AI 遵守度和減少 context 負擔 | 化解「LLM-native 格式值得嗎」質疑 |
| 6 | 使用場景：開發者收到規格書（模糊或明確），讀後提供給 AI 開發 | 暴露 Phase 1 缺少需求澄清步驟的問題 |

### 選 A 後浮現的新問題

| # | 新問題 | 說明 |
|---|--------|------|
| 1 | Phase 0 需重寫 | 移除所有 plugin 偵測後，Phase 0 剩餘內容太少，可與 P1 合併 |
| 2 | TDD 紀律需內建 | 不再依賴 superpowers TDD skill，需自行維護 RED-GREEN-REFACTOR 規則 |
| 3 | 並行度瓶頸 | 每個任務內部 test→work→qa 是序列的，test-agent（Opus）完成速度限制了真正的並行度 |

| 5 | Phase 1 缺需求澄清 | 直接從「讀規格→拆任務→寫 PLAN」，如果規格模糊，TL 基於錯誤理解拆任務後使用者要推翻重來 |
| 6 | DELIVERY 的 Agent Metrics 與 audit trail 無關 | 跟技能開發者的 debug 需求有關，不該混入使用者的交付報告 |

## 四、方案比較與決策

### 4.1 外部 plugin 耦合：三方案比較

| 方案 | 做法 | 代價 |
|------|------|------|
| **A：完全獨立** | 移除所有偵測邏輯，把 TDD、verification 等最佳實踐直接內建 | 維護成本低，但可能與 superpowers 重複 |
| **B：硬耦合** | 宣告 superpowers 為 peer dependency，要求使用者必須安裝 | 測試矩陣縮小，但限制使用者自由度 |
| **C：介面契約** | 定義 dev-team 期望的「TDD discipline interface」，superpowers 可為其中一個實作 | 最乾淨但最多工作 |

**決策：選 A — 完全獨立。**

理由：
- 消除測試矩陣爆炸問題（從 16 種組合降為 1 種）
- TDD 的 RED-GREEN-REFACTOR 是穩定知識，不會有版本漂移，一次性內建成本可忽略
- 選 A 後拆分 repo 的邏輯更自洽 — 既然完全不依賴 TF-Skills，就不需要同一個 repo

### 4.2 Phase 合併：P0 + P1

**決策：合併為 Phase 0: Project Understanding + Task Planning。**

整合後的流程：
```
Phase 0 (新)
├── 掃描專案結構（原 P0 的有用部分）
├── AskUserQuestion: 規範資料夾路徑？（原 .standards/ 自動偵測 → 改為主動詢問）
├── 讀規格書 → 輸出理解摘要 → AskUserQuestion 確認理解（新增：需求澄清）
└── 確認後 → 拆任務 → 寫 PLAN → AskUserQuestion 確認 PLAN（原 P1）
```

理由：
- 移除 plugin 偵測後，原 P0 內容太少，不值得獨立為一個 Phase
- 需求澄清步驟自然放在任務規劃前
- Phase 編號從 P0-P5 簡化為 P0-P4

### 4.3 TDD 紀律內建

**決策：直接寫進 test-agent prompt，自行維護。**

內建內容：
- 寫最小化測試驗證 acceptance criterion
- 每個測試必須可執行且必須失敗（沒有 implementation 存在時）
- 執行驗證：跑一次確認全部 fail（missing module、missing function，不能是 syntax error）
- 如果測試在沒有 implementation 時就 pass → 測試有問題，刪除重寫
- RED-GREEN 語義：test-agent 的測試是 RED，Worker 讓它們變 GREEN

理由：
- TDD 是穩定知識，不會因版本漂移而需要頻繁更新
- 維護成本極小，比依賴外部 skill 的風險更低

### 4.4 Agent Metrics 移除

**決策：從 DELIVERY template 移除 Agent Metrics section（8 section → 7 section）。**

理由：
- DELIVERY 的目的是 code review audit trail（使用者期望 4）
- Agent Metrics（token 花費、時間）跟 code review 無關
- 技能開發期間需要看花費數據可用 ccusage 工具
- Agent 層級的 log 中仍保留 `[METRICS]` 行作為 debug 資訊

### 4.5 Scope 自驗（附帶改進）

**決策：Worker 完成前必須跑 `git diff --name-only` 比對 ALLOWED list。**

流程：
```
Worker COMPLETION SEQUENCE:
1. Verify all tests pass
1.5. Run `git diff --name-only` → 比對 ALLOWED list
     超出範圍 → git checkout -- {file} 回退 → 記 log → 繼續
2. Write final log entries
3. SendMessage TL
4. Wait for shutdown
```

理由：
- 前置驗證比事後審查可顯著降低重做成本
- 在 Worker 層面攔截 scope 違規，不需要等 QA 抓再重做

### 4.6 Execution Strategy 參數（附帶改進）

**決策：不立即實作，僅在設計中留下擴展空間。**

```
目前預設：parallel-per-task（每個任務獨立跑 test→work→qa）
未來可選：batch-test-first（所有測試一次寫完 → 所有 worker 同時開始）
```

理由：
- 沒有 SKILL 就沒法測，是雞生蛋問題
- 先用 `parallel-per-task` 上線，日後有基線數據再 A/B 測試
- 設計成可調參數，方便日後比較時間/token/品質

### 4.7 `.standards/` 偵測

**決策：保留偵測，但改為 AskUserQuestion 主動詢問路徑。**

理由：
- `.standards/` 是專案層級的慣例，不屬於任何特定 skill
- 偵測它就像偵測 `tsconfig.json` 一樣自然
- 但自動偵測有誤判風險，改為主動詢問更可靠

## 五、使用者整理的四大議題與共識

使用者將所有問題收斂為四個議題，雙方達成共識：

### 議題 1：P0/P1 整合 + 需求澄清 + 規範偵測

共識：三件事合併成一個「啟動階段」完全合理。Phase 編號 P0-P5 → P0-P4。小幅變更。

### 議題 2：TDD 紀律內建

共識：TDD 的 RED-GREEN-REFACTOR 是穩定知識，不會有版本漂移問題。
一次性寫進 test-agent prompt，維護成本可忽略。**結案。**

### 議題 3：Agent Metrics 處理

使用者觀點：技能開發階段需要看花費數據來調整設計，但用 ccusage 就能處理。
結論：DELIVERY 只服務於 code review audit trail，不混入 skill 開發者的 debug 需求。
**直接移除。**

規範資料夾議題併入議題 1（改為 AskUserQuestion 詢問路徑）。

### 議題 4：並行度 vs 實證

共識：
- 增加並行度的優點是時間，期望用更多花費換時間和兼顧品質
- 沒有 SKILL 就沒法做實證比較測試
- 解法：在 SKILL 中設計成可調參數（execution strategy），先用預設模式上線，
  日後有基線數據再測 batch-test-first 等替代方案

## 七、設計思維總結

本次審查的核心洞察：

1. **「可選偵測」是最差的設計模式** — 看似靈活，實際上測試不可能涵蓋所有組合，且迴避了耦合決策。
   選 A（完全獨立）後，設計變得乾淨可測。

2. **文件至上的哲學支撐了看似冗餘的設計** — PLAN.md、DELIVERY.md 表面上是重複資訊，
   但在「不信任 LLM context/memory」的前提下，它們是 agent 間溝通的唯一可靠媒介。

3. **前置驗證 vs 事後審查的成本差異顯著** — Worker scope 自驗應該在 Worker 層面攔截，
   不是等 QA 抓到再重做。

4. **設計應為未來的實證留空間** — 並行度策略設計成可調參數，先上線收集基線數據，
   再決定是否切換到 batch-test-first 模式。

5. **穩定知識可以內建，不穩定知識才需要外部依賴** — TDD 的 RED-GREEN-REFACTOR 是穩定的，
   直接內建比依賴外部 skill 的版本同步更可靠。
