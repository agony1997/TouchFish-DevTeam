## 二、架構設計質疑

### 2.1 [x] Token 成本透明度

**嚴重度：High** | 合併 5.1

**問題：** 5 任務完整執行 spawn ~18 個 agent（7 個 Opus）。使用者無法預知規模和成本。

**決策：** Phase 0 確認 PLAN 時輸出預估 spawn 數量和模型分佈（例如「預計 3 Opus + 8 Sonnet」），讓使用者知道規模後再決定是否執行。

**實作：** 修改 SKILL.md Phase 0 步驟 12（AskUserQuestion 確認 PLAN 時加入預估資訊）。

### 2.2 [x] QA-task 改用 Opus

**嚴重度：Medium**

**問題：** 三方交叉驗證 + 標準合規 + 程式碼品質 + 檔案範圍 = 6 步驟複雜推理，品質閘門應用最強模型。

**決策：** qa-task 從 Sonnet 改為 Opus。

**實作：** 修改 `prompts/qa-task.md` spawn config 和 SKILL.md 中的團隊架構表。

### 2.3 [x] Fix Task context 斷裂

**嚴重度：High**

**問題：** QA FAIL → 新 Worker（新 context）。新 Worker 不知道前任為什麼 fail。

**決策：** TL 建立 fix task 時，將 QA 失敗原因摘要（人類可讀）寫入 task description。TL 本來就有 QA 回報結果，只是改變傳遞方式。

**實作：** 修改 SKILL.md Phase 2 步驟 2（FAIL 處理流程加入摘要步驟）。

### 2.4 [x] qa-global 加可選整合測試

**嚴重度：Medium**

**問題：** Phase 3 只做靜態分析，缺乏整合驗證。

**決策：** qa-global prompt 加入可選步驟：如果專案有現成測試框架，跑一次全部測試確認無回歸。

**實作：** 修改 `prompts/qa-global.md` 加入 CHECK 6。

### 2.5 [x] 3 Worker 上限標註暫定

**嚴重度：Low**

**問題：** `≤ 3 Workers concurrent` 無依據。

**決策：** 標註為暫定經驗值，待實測數據調整。

**實作：** 修改 SKILL.md Phase 2 步驟 1 加入註記。

### 2.6 [x] METRICS 改由 TL 記錄

**嚴重度：Low**

**問題：** Sub-agent 自己取不到 token 消耗，但 TL 可從 Agent tool 回傳取得。Worker（teammate）確實取不到。

**決策：** Sub-agent prompt 移除自報 [METRICS] 要求。TL 從 Agent tool 回傳取得 sub-agent 的 usage 後寫入 tl.log。Worker 的 [METRICS] 保留 n/a（客觀限制）。

**實作：** 修改所有 sub-agent prompts（test-agent、qa-task、qa-global、delivery-sub）移除 [METRICS] 自報；修改 SKILL.md 加入 TL 記錄 sub-agent metrics 的步驟；修改 `references/log-templates.md` 對應調整。

## 三、Prompt 工程質疑

### 3.1 [-] Worker scope 自驗只檢查檔案級別

**嚴重度：Medium**

**不處理理由：** 檔案級自驗（Worker）+ 行為級審查（QA STEP 3 scope creep 檢查）= 兩層防禦已足夠。加強自驗需要 Sonnet 做模糊判斷，誤判風險高且與 QA 重複。

### 3.2 [x] test-agent TDD DISCIPLINE 補完

**嚴重度：Low**

**決策：** 在 test-agent prompt RULES section 加入兩行：
1. 需求模糊到無法寫可驗證斷言 → 停止，回報 TL 澄清
2. 同一 acceptance criterion 可多個斷言（不強制 single assertion）

**實作：** 修改 `prompts/test-agent.md` RULES section。

### 3.3 [-] Communication Discipline 對「pure acknowledgment」判定模糊

**嚴重度：Low**

**不處理理由：** Worker 的規則按優先序判斷（Instruction → Complete → Problem → Acknowledgment），只有前三條都不匹配才走第四條。誤判不回的機率極低，且影響可忽略。
